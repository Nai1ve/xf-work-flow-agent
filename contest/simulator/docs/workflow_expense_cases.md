8. 费用类验证集（`beta_wf_0231 ~ beta_wf_0260`）
- 共 `30` 题
- `submitted`：11 题
- `draft_saved`：9 题
- `blocked`：10 题

分组：
- `beta_wf_0231 ~ 0240`
  有效提交 / 草稿题，强调自然语言理解、多明细、数量单价换算，但刻意避开训练集模板措辞
- `beta_wf_0241 ~ 0250`
  语义歧义题，强调“错选也合法，但必须选语义最正确的项目 / 大类 / 小类”
- `beta_wf_0251 ~ 0260`
  阻塞拒绝题，强调“信息不足时不能乱保存、不能直接提交”

设计原则：
- 不直接复用训练集里的“产品发布会 + 明确子类 + 预算直给”模板
- 增加近义项目、近义类别、近义子类，考验 Agent 泛化能力
- 增加必须阻塞的 case，避免只会硬填表单的策略拿高分
- 成功题不只看最终字段正确，也会逐步收紧到要求走过正确查询路径：
  - `workflow.project_search`
  - `workflow.browser_search(field_id=29023)`
  - `workflow.browser_search(field_id=29028, dep=...)`
- 阻塞题不鼓励“一上来就 blocked”：
  - 至少应先查过项目
  - 通常还应查过大类
  - 能唯一化时继续保存/提交，不能唯一化时再阻塞

这些 case 统一覆盖以下能力：
- `workflow.catalog`
- `workflow.schema`
- `workflow.project_search`
- `workflow.browser_search`
- `workflow.save`

并混合考察以下变化维度：
- `project_name` 搜索与 `project_code` 直查
- `draft` 与 `submit`
- 单明细与多明细
- 数量、单价、金额的组合填写
- 不同真实大类和真实子类编码
- 项目歧义、大类歧义、子类歧义
- 显式要求先做依赖查询，再保存/提交
- 错误路径也合法时仍需选择语义最正确的项目或子类
- 信息不足时必须阻塞，不能乱保存或直接提交

评分口径补充：
- 成功题会优先检查最终 `draft_saved / submitted` 状态与主表、明细字段是否正确。
- 一批代表性费用题还会显式要求调用过正确的查询路径，而不是仅靠最终编码“蒙对”。
- `blocked` 题通常会同时要求：
  - 调用过必要的前置查询
  - `未调用过 workflow.save`
  - `final_answer` 返回 `status=blocked`
